# EstateOS — Production Audit

**Date:** 2026-06-04
**Scope:** Full application (`C:\Users\HP\Desktop\Realestate saas`) — Next.js 16 / React 19, Clerk auth, Prisma + Supabase Postgres, Paystack, Cloudflare R2, Upstash Redis, Inngest, Sentry, PostHog, Twilio, Resend, Mapbox.
**Size reviewed:** 621 source files, ~92,700 LOC, 95 API routes, ~105 co‑located test files, 75+ Prisma models.
**Reviewers (composite):** Product, Senior UI/UX (Linear/Uber/Apple lens), Security, Full‑Stack, SaaS Architecture.
**Status:** Audit only. No code was changed. Awaiting approval before any implementation.

## Executive Summary

The honest version: **this is a genuinely well‑engineered application — well above the typical "one company is using it in production" codebase.** Whoever built this understands multi‑tenant SaaS. The security model is mature, the tenancy isolation is deliberate and centralized, validation is everywhere, there are ~105 tests, structured logging with credential redaction, health/readiness endpoints, environment validation via Zod, and signed‑and‑verified webhooks. Most "production SaaS audits" find missing auth checks and SQL injection; this one does not.

The real risks are **not** in the core security primitives. They are in three places:

1. **Environment hygiene / process risk.** Local development currently points at the **production database** (`.env.local` has `DATABASE_URL` → the live Supabase project) while also running with `NODE_ENV=development` and `ESTATEOS_ENABLE_DEV_BYPASS=true`. A safety guard blocks seed/destructive scripts, but normal app CRUD through the UI is not blocked — and dev‑bypass means that local server has unauthenticated admin/superadmin access to **live tenant data**. This is the single most important thing to fix. The team already knows (the README has a "Temporary Production Database Safety Guard" section), but "temporary" needs to end before a second company is onboarded.

2. **Mobile/responsive experience.** Your instinct is correct. The shared dashboard shell (used by both admin and buyer portals) only becomes a real sidebar at 1024px. Below that, all 21 admin nav links render as a stacked tile grid *above* the page content, with no hamburger, drawer, or sticky top bar. On a phone, an agent scrolls past ~11 rows of navigation before reaching the actual screen. Additionally, the app's fonts ("Aptos", "Baskerville Old Face") are Windows‑only and are not web‑loaded, so on Mac/iOS/Android/Linux the typography silently falls back and looks off‑brand.

3. **Operational gaps that bite at scale, not at N=1.** Rate limiting exists on only 3 endpoints (payments, uploads, and KYC are unthrottled). There's no Content‑Security‑Policy header. The dashboard shell re‑runs ~6 sequential queries on every authenticated page load with no caching. Realtime is in‑memory (won't fan out across serverless instances) but is currently disabled, so it's a "before you turn it on" issue.

**Is it safe to keep running in production for the current single company?** Yes — with the caveat that the shared dev/prod database is an accident waiting to happen and should be separated this week. **Is it ready to onboard more companies?** Almost, but fix the environment separation, rate limiting, and mobile navigation first.

## Critical Security Findings

> Overall security posture is **strong**. The findings below are ranked by severity; note that the highest‑severity item is operational, not a code vulnerability.

### CRITICAL — Shared production database in local development with dev‑bypass

- **Files:** `.env.local` (`DATABASE_URL` resolves to host `aws-0-eu-west-1.pooler.supabase.com`, the production Supabase project; `NODE_ENV="development"`; `ESTATEOS_ENABLE_DEV_BYPASS="true"`), `src/lib/db/production-db-guard.ts`, `src/lib/auth/session.ts` (`resolveDemoSessionRole` returns `superadmin`/`admin` unconditionally when `allowDevBypass` is on), `README.md` → "Temporary Production Database Safety Guard".
- **Explanation:** `buildFeatureFlags` correctly hard‑gates `allowDevBypass` to non‑production, so this can never leak to the deployed app — that part is well done. The problem is the **developer's local environment**: with dev‑bypass on, `getAppSession` mints a superadmin/admin session from a cookie with **no authentication**, and that session reads/writes the **live** database. `assertProductionDatabaseWriteAllowed` only guards seed/cleanup/sample scripts and demo session bootstrap — it does **not** wrap ordinary CRUD performed through the app UI/API. So a local dev server (or anyone who can reach it) effectively has unauthenticated, full‑tenant superadmin access to production data.
- **Risk:** Critical (accidental data corruption of a live customer; no isolated environment to test changes; a single mistake hits real buyers' KYC/payment records).
- **Recommended fix:** Stand up a separate dev/staging Supabase project. Point `.env.local` at it. Keep `ESTATEOS_ENABLE_DEV_BYPASS` only in environments that cannot reach production data. Retire the "temporary" shared‑DB arrangement before onboarding company #2.

### HIGH — Rate limiting covers only 3 endpoints

- **Files:** `src/lib/rate-limit.ts` (only `inquiryRateLimit`, `inspectionRateLimit`, `supportRateLimit`). Used in `src/app/api/inquiries/route.ts`, `src/app/api/inspections/route.ts`, `src/app/api/portal/support/route.ts`. **Unprotected:** `src/app/api/payments/initialize/route.ts`, `src/app/api/payments/verify/route.ts`, `src/app/api/uploads/sign/route.ts`, `src/app/api/uploads/documents/route.ts`, `src/app/api/reservations/route.ts`, `src/app/api/saved-properties/route.ts`, `src/app/api/portal/kyc-submissions/route.ts`, `src/app/api/accept-invitation/[token]/route.ts`, and all `api/admin/*` mutations.
- **Explanation:** Upstash is already wired; the limiters just aren't applied broadly. `payments/initialize` calls Paystack and generates DB rows; `uploads/sign` mints R2 signed URLs; KYC/invitation endpoints touch sensitive flows. All are abusable for cost amplification, enumeration, or brute force.
- **Risk:** High (financial abuse via payment init spam, R2/email/SMS cost amplification, invitation‑token brute force).
- **Recommended fix:** Add per‑user + per‑IP limiters (you already have the sliding‑window pattern) to all state‑changing and cost‑incurring endpoints. Prioritize payments, uploads, accept‑invitation, KYC.

### MEDIUM‑HIGH — No Content‑Security‑Policy

- **File:** `next.config.ts` (`headers()` sets only `Referrer-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, `Permissions-Policy`).
- **Explanation:** No CSP means any injected script (including via tenant‑customizable branding surfaces) executes freely. The basics are present but the strongest XSS mitigation is missing.
- **Risk:** Medium‑High (XSS impact amplification; tenant branding increases the input surface).
- **Recommended fix:** Add a CSP (start in `Content-Security-Policy-Report-Only` to avoid breaking Clerk/Mapbox/PostHog/Sentry, then enforce). Allowlist the known third‑party origins; use nonces for inline scripts.

### MEDIUM — Production database identifier committed to the repo

- **Files:** `src/lib/db/production-db-guard.ts` (`ESTATEOS_PRODUCTION_PROJECT_REF = "epxbejutuodmnsdfvcjr"`, `ESTATEOS_PRODUCTION_DATABASE_HOST = "aws-0-eu-west-1.pooler.supabase.com"`), `.env.example` (same values).
- **Explanation:** A Supabase project ref is an identifier rather than a secret, but committing it pinpoints exactly which host holds production data. Combined with the shared‑DB practice, it lowers the bar for an attacker who gets any code access.
- **Risk:** Medium.
- **Recommended fix:** Move these values to environment variables (the guard already reads `PRODUCTION_DATABASE_PROJECT_REF`/`PRODUCTION_DATABASE_HOST` as overrides — make them required and drop the hardcoded defaults). Scrub from `.env.example`.

### LOW — Non‑constant‑time secret comparison for cron

- **File:** `src/app/api/internal/automation/run/route.ts` (`header !== env.CRON_SECRET`).
- **Risk:** Low (timing side channel; the secret is high‑entropy and the endpoint is internal).
- **Recommended fix:** Use `crypto.timingSafeEqual`.

### Notable security strengths (verified, keep these)

- **Tenant isolation is centralized and correct:** `scopeTenantWhere` / `scopeTenantQueryArgs` inject `companyId`, `rejectUnsafeCompanyIdInput` blocks caller‑supplied company IDs, `assertTenantAccess` enforces cross‑tenant denial, and `selectAuthenticatedCompany` deliberately ignores host/hint companies for authenticated users (prevents host‑header tenant takeover). (`src/lib/tenancy/db.ts`, `src/lib/tenancy/context.ts`.)
- **IDOR protection on downloads:** `src/app/api/documents/[documentId]/download/route.ts` checks both `assertDocumentAccess` (tenant + buyer ownership) **and** storage‑key namespace match before issuing a signed URL. Receipts/payments verify tenant ownership (`payments/verify` rejects mismatched `userId`).
- **Webhooks are verified:** Paystack HMAC signature (`verifyPaystackSignature`) and Clerk Svix signature both checked before processing; both reject on failure.
- **Superadmin is double‑gated:** requires `SUPER_ADMIN` role **and** email allowlist **and** `clerk` mode in production (`src/lib/auth/superadmin.ts`); claims‑sourced superadmin is stripped.
- **Central middleware:** `src/proxy.ts` (Next 16 middleware) runs `clerkMiddleware` + `auth.protect()` for `/admin`, `/portal`, `/superadmin` in production; API routes are individually guarded with `requireAdminSession`/`requirePortalSession`/`requireSuperAdminSession` (consistently applied across the 95 routes sampled).
- **Dev tooling self‑gates:** `DevAccessSwitcher` returns `null` unless `allowDevBypass`; `/api/dev/*` routes return 403/404 outside development. Confirmed safe in production.
- **Secrets:** `.env*` is gitignored (only `.env.example` is tracked); logger redacts DB credentials from error messages/stacks.

## UI/UX & Responsiveness Findings

### 1. Dashboard shell mobile navigation (the big one)
- **Component:** `src/components/portal/dashboard-shell.tsx` (shared by admin **and** portal).
- **What's wrong:** Layout is `lg:grid-cols-[260px_minmax(0,1fr)]` — a sidebar only at ≥1024px. The nav uses `grid-cols-2 sm:grid-cols-3 lg:block`, so on phones/tablets the **21 admin links** (or 13 portal links) render as a tile grid stacked above the page content. There is no hamburger, no slide‑in drawer, and no sticky mobile header. Tablet (640–1023px) gets the same stacked grid.
- **How to improve:** Add a sticky top bar on `<lg` with a logo + menu button that opens a slide‑in drawer (a `Sheet` primitive). Keep the desktop sidebar as‑is. Collapse 21 links into ~5 grouped sections with disclosure. This single change fixes the most‑felt pain.

### 2. Typography doesn't load on most devices
- **File:** `src/app/globals.css` (`--font-sans: "Aptos", "Segoe UI", …`; `--font-serif: "Baskerville Old Face", …`). No `next/font` usage anywhere; no self‑hosted fonts in `public/`.
- **What's wrong:** "Aptos" and "Baskerville Old Face" ship with Microsoft Office/Windows. On macOS, iOS, Android, and Linux (i.e., most buyers, especially on mobile) they don't exist, so text silently falls back to Helvetica/serif default. The product looks different — and less premium — on the majority of devices.
- **How to improve:** Adopt `next/font` with a self‑hosted or Google‑hosted family (e.g., Inter/Geist for sans, a real serif for display), or properly license + self‑host the brand fonts. This is a quick, high‑leverage polish win.

### 3. Missing shared primitives → inconsistent interactions
- **Folder:** `src/components/ui/` has only `avatar, badge, button, card, input, separator, textarea`.
- **What's wrong:** No shared `Dialog/Modal`, `Select`, `Dropdown`, `Tabs`, `Tooltip`, `Table`, or `Sheet`. Modals like `src/components/admin/invite-member-modal.tsx` are one‑offs. With 228 components, bespoke re‑implementation means subtly different focus traps, escape handling, animations, and ARIA across screens.
- **How to improve:** Introduce a primitive layer (Radix UI pairs well with your token system), migrate bespoke modals/menus/tables onto it. This also closes accessibility gaps for free.

### 4. Button lacks a loading/pending state
- **File:** `src/components/ui/button.tsx`.
- **What's good:** Variants via `cva`, proper disabled styling, focus‑visible ring via `.admin-focus:focus-visible` (in `globals.css`), tenant‑themable radius.
- **What's wrong:** No built‑in `isLoading`/`aria-busy`/spinner, no `type` default. Async actions across the app each handle pending state ad hoc (or not at all), so submit buttons can be double‑clicked and give no feedback.
- **How to improve:** Add `loading` + `leftIcon` props with a spinner and `aria-busy`, default `type="button"`, and a subtle `:active` press state for premium feel.

### 5. Loading / error / empty states are uneven
- **Counts:** 7 `loading.tsx` across ~110 route segments, 10 `error.tsx`, 1 `not-found.tsx`, **0 `global-error.tsx`**.
- **What's wrong:** Many `force-dynamic` pages show nothing during data fetch (no skeletons), and there's no root error boundary — if the root layout/providers throw, the user sees Next's unstyled default. Empty states exist as CSS (`.admin-empty-surface`) but coverage isn't uniform.
- **How to improve:** Add `global-error.tsx`; add `loading.tsx` skeletons to the heavy pages (Deal Board, Transactions, Analytics, Listings); standardize an `EmptyState` component.

### 6. Dead dark‑mode dependency
- `next-themes` is in `package.json` but there is no `ThemeProvider` and no `.dark` tokens in `globals.css`. Either finish dark mode or remove the dependency.

### What's genuinely good (don't lose it)
- Cohesive design‑token system (brand/ink/sand palettes, spacing, radii, motion duration).
- `prefers-reduced-motion` handling, skeleton shimmer, print/`@page` styles, tenant theming, tasteful micro‑motion (`tenant-fade-up`, nav active indicator). The *foundation* is good — the gaps are responsiveness, fonts, primitives, and state coverage.

## Product & User Flow Findings

- **Strong core model.** The buyer funnel (Inquiry → Inspection → Reservation → Transaction/Installments → Payment → Receipt → Contract) is clearly modeled end‑to‑end, with an onboarding checklist and sample‑data flow. This is the product's spine and it's solid.
- **Admin surface sprawl / overlapping concepts.** 21 top‑level admin sections (`src/components/portal/dashboard-shell.tsx` → `adminLinks`): Deal Board, Payments, Clients, Leads, Transactions, Pipeline, Analytics, Benchmarks, Feasibility, Contracts, Listings, Team, Marketers, Bookings, Assets, Billing, Settings, Documents, Testimonials, Notifications, Audit Logs. Several describe the *same* funnel from different angles — **Deal Board vs Leads vs Clients vs Pipeline vs Transactions** — and **Analytics vs Benchmarks** overlap. A new admin won't know where to take an action. Recommend grouping into ~5–6 buckets (Sales/CRM, Listings, Finance, Content, Team, Settings) with secondary nav, and clarifying the distinction (or merging) of the funnel views.
- **Feasibility / development‑calculations is heavy.** A sophisticated module (sales mix, phases, AI narrative, decision workflow). Great for developers selling off‑plan; likely overkill for a small brokerage. Gate it by plan/role so core CRM users aren't overwhelmed.
- **Two marketing surfaces.** `(marketing)` (tenant public site) and `(platform-marketing)` (EstateOS sales site) coexist. This is correct architecturally but make sure tenants never accidentally land on platform pages and vice‑versa (the tenant resolution logic handles this; worth an explicit QA pass).
- **Before onboarding more companies:** tighten the admin information architecture, ensure plan‑based gating hides modules a tenant didn't buy, and pressure‑test the self‑serve onboarding (sample workspace → first listing → first inquiry) on mobile.

## Code Quality & Architecture Findings

**Overall: high quality.** Clear separation: `src/lib` (cross‑cutting: auth, tenancy, payments, storage, validations, ops), `src/modules` (domain services, ~43 modules), `src/app` (routes), `src/components`. Zod validation throughout, `cva` for variants, ~105 co‑located tests, centralized guards and tenancy helpers. This is maintainable.

Specific items:

- **`src/modules/admin/deal-board.ts` is 1,096 lines.** It's well‑written and batches queries, but it mixes data access, scoring, formatting, and view assembly. Split into `queries`, `scoring/risk`, and `view-model` files for maintainability.
- **`DashboardShell` does heavy work in a server component on every render** (`src/components/portal/dashboard-shell.tsx`): `requireAdminSession`/`requirePortalSession` → `getTenantPresentation` → `getAppSession` → `resolveBuyerTenantContextForKyc` → a raw `$queryRaw` for the portal user → `notification.count`. Several are sequential and could be `Promise.all`'d; none are cached. This runs for *every* admin/portal page navigation.
- **Inconsistent data access.** Almost everything uses the Prisma ORM with scoped helpers, but `DashboardShell` drops to `prisma.$queryRaw` for the portal user (parameterized, so safe — but inconsistent with the rest of the codebase).
- **Misleading helper signature.** `src/lib/tenancy/authenticated-company.ts` → `selectAuthenticatedCompany` takes three args and returns only `sessionCompany`. The behavior is intentional (security), but the signature invites misuse; add a comment or simplify.
- **Route‑handler boilerplate is repetitive.** 95 routes repeat the same `try { requireX } catch { return fail(401) } → parse → scope → mutate → fail(400)` shape. A shared `withTenant(handler, { area, roles })` wrapper would cut drift risk and make it impossible to forget a guard.
- **`force-dynamic` on the admin layout** (`src/app/(admin)/admin/layout.tsx`) disables static optimization for the entire admin area. Necessary for per‑tenant auth, but it means zero caching — see Performance.

## Performance & Scalability Findings

**Where it's good:**
- **No obvious N+1 in the main dashboard.** `deal-board.ts` batches ~12 reads via `Promise.all` and maps results in memory (no per‑row queries). Selects are scoped and narrow.
- **Correct serverless DB topology.** `DATABASE_URL` uses the Supabase transaction pooler (pgbouncer, port 6543), `DIRECT_URL` for migrations; `config.ts` actively validates this (`getDatabaseTopologyIssues`). This is the right setup for Vercel.
- **Background work via Inngest** (`src/inngest/functions.ts`, `/api/inngest`, cron via `/api/internal/automation/run`) — good for offloading.
- **Monitoring in place:** Sentry + PostHog server/client exception capture with sampling.

**What will hurt as you go from 1 → 10 → 50 → 100 companies:**
- **Per‑page query fan‑out.** Every authenticated page re‑resolves session + tenant + branding + buyer profile + notification count with no caching (`DashboardShell`). At 100 companies with concurrent users, that's a multiplier on DB load and latency on *every navigation*. **Fix:** `unstable_cache` the tenant branding/presentation keyed by `companyId`, cache the unread‑notification count briefly, and parallelize the shell's awaits.
- **Realtime won't scale.** `src/app/api/realtime/stream/route.ts` + `subscribeRealtimeEvents` use an in‑process emitter — events only reach clients on the *same* serverless instance. It's currently disabled (`REALTIME_TRANSPORT=polling`), so this is a "before you enable SSE" blocker. **Fix:** Redis pub/sub (you already have Upstash) or a hosted realtime (Supabase Realtime / Pusher / Ably).
- **Tenant images likely unoptimized.** `next.config.ts` `remotePatterns` allows only `images.unsplash.com`; R2 isn't listed. `next/image` is used in just 6 files. Tenant property photos served from R2 are therefore probably not run through `next/image` (or use the public base directly), meaning full‑size payloads on listing/gallery pages. **Fix:** add the R2 public host to `remotePatterns` and route tenant media through `next/image` (or an image‑resizing layer).
- **Heavy client libraries.** `mapbox-gl` and `recharts` are large. Ensure they're dynamically imported only on the screens that use them (maps, analytics) so the base bundle stays light.
- **No caching on the admin area** (`force-dynamic`). Expected for authed dashboards, but combine with per‑page query caching above to keep DB pressure linear, not multiplicative.

## Recommended Fix Roadmap

### Fix immediately — before onboarding another company
1. **Separate environments.** Create a dedicated dev/staging Supabase project; repoint `.env.local`; ensure `ESTATEOS_ENABLE_DEV_BYPASS=true` only exists where it cannot reach production data. End the shared‑DB arrangement. *(Finding: Critical #1)*
2. **Add rate limiting** to `payments/initialize`, `payments/verify`, `uploads/sign`, `uploads/documents`, `accept-invitation/[token]`, `reservations`, `portal/kyc-submissions`, and admin mutations. *(High #3)*
3. **Fix mobile navigation** in `dashboard-shell.tsx` (sticky top bar + slide‑in drawer below `lg`). *(High #2)*
4. **Add a CSP** (report‑only first, then enforce). *(Medium‑High #4)*
5. **Confirm Supabase PITR/backups** are on and do one test restore.
6. **Move the production DB ref/host out of source** into required env vars; scrub `.env.example`. *(Medium)*

### Fix within 1–2 weeks
7. **Web‑load fonts** via `next/font` (fix cross‑device typography). *(Medium‑High #5)*
8. **Add `global-error.tsx`** and `loading.tsx` skeletons for Deal Board, Transactions, Analytics, Listings. *(Medium #7)*
9. **Cache + parallelize the dashboard shell** (`unstable_cache` branding/presentation by `companyId`; brief cache on notification count; `Promise.all` the awaits). *(Medium #6)*
10. **Add `loading`/`aria-busy` to `Button`** and a default `type`. *(UX #4)*
11. **Consolidate admin nav** into grouped sections; clarify or merge Deal Board / Leads / Clients / Pipeline / Transactions. *(Product #9)*

### Fix within 30–60 days
12. **Introduce a primitive layer** (Dialog, Select, Tabs, Table, Tooltip, Sheet — e.g., Radix) and migrate bespoke components; closes a11y gaps. *(UX #8)*
13. **Responsive pass** across the ~134 components with no breakpoints (tables → stacked cards on mobile, form layouts, modals).
14. **Replace in‑memory SSE with Redis pub/sub** (or hosted realtime) before enabling SSE. *(Scalability #10)*
15. **Route tenant media through `next/image`** (add R2 host to `remotePatterns`); dynamically import Mapbox/Recharts.
16. **Add a shared route‑handler wrapper** (`withTenant`) to standardize auth/validation/error and prevent guard drift.
17. **Constant‑time cron secret compare**; remove or finish `next-themes`.

### Long‑term improvements
18. **Plan‑based feature gating in the UI** so smaller tenants don't see Feasibility/Benchmarks they didn't buy.
19. **Postgres Row‑Level Security** as defense‑in‑depth beneath the (already good) app‑layer tenant scoping.
20. **Design‑system docs + Storybook**, and a formal **WCAG accessibility audit**.
21. **Operational maturity:** log drain/retention, SLOs, incident runbook/on‑call, and a 50‑company load test with index review.
